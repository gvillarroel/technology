# ADR 0007: Documentation Metadata Must Be Auto-Derived

## Status

Accepted

## Context

The `documents` experience is repository-driven. Its YAML configuration exists to tell the site
which repositories to inspect and which Markdown paths should be published. That configuration is a
selection mechanism, not a metadata source of truth.

The current implementation allowed extra descriptive metadata such as repository owner to be
declared in `data/document-repositories.yaml` and then shown in the UI as if it were discovered
fact. The same experience also rendered fallback strings such as `Timestamp unavailable` when no
automatic timestamp could be resolved.

That mixes two different kinds of information:

- selection rules declared by site maintainers
- metadata that the collection processor can derive from scanned files or repository history

When those are mixed, the UI implies confidence that does not exist. A label shown in the
documentation library should represent either scanned content or an automatically derived repository
signal, not a manual annotation hidden inside the repository-selection config.

## Decision

For the repository-driven `documents` library:

- `data/document-repositories.yaml` is limited to repository discovery and file selection concerns
- YAML entries must not carry manual display metadata such as owner labels for documentation pages
  or repositories
- documentation UI and Markdown outputs may show metadata only when the collection processor
  derives it automatically
- if an automatic metadata field cannot be derived, that field is omitted from the UI instead of
  showing a placeholder such as `Timestamp unavailable`

Accepted automatic sources include:

- document frontmatter in scanned Markdown files
- repository commit history or file commit dates resolved during collection
- aggregate values computed from scanned pages, such as a repository latest update

## Consequences

### Positive

- metadata shown in `documents` has clear provenance
- repository-selection config stays focused and easier to maintain
- users are less likely to mistake manual config labels for scanned repository facts
- HTML and Markdown representations stay aligned on the same evidence standard

### Negative

- some repository cards and detail pages will display less metadata when scanned sources do not
  provide it
- teams that want richer metadata must add it to scanned documents or extend the processor with a
  real automatic source

## Implementation Guidance

- remove manual documentation metadata fields from `data/document-repositories.yaml`
- treat document-level frontmatter as valid only when it comes from the scanned Markdown source
- omit document or repository owner fields unless the processor can derive them from scanned pages
- omit update labels when no automatic timestamp is available
- keep scan-specific operational facts such as branch and last scan time when those are generated by
  the processor itself

## Validation

This decision is implemented by:

- removing manual owner metadata from `data/document-repositories.yaml`
- updating the document collection pipeline to stop depending on YAML owner fields
- suppressing empty owner and update labels in both HTML and Markdown document outputs

## Decision Summary

Repository-driven documentation may expose only auto-derived metadata. Selection YAML chooses what
to scan; it does not define what the UI claims as document metadata.
